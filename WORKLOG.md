# 公安知識庫 — 工作紀錄與工作計畫

---

## 系統背景

本系統為公共安全檢查知識查詢平台，目標是**讓新進人員能快速查到正確的公安申報知識**。

- **第一階段（已完成）**：封閉資料庫，建立於 Ragic，供內部人員填寫與查閱
- **第二階段（進行中）**：本系統，結構化知識卡片 + 智慧搜尋，部署於 Vercel

## 二、決策紀錄

| 議題 | 決定 |
|------|------|
| 找不到答案時 | 明確顯示「找不到」，**不做 AI 推斷**，避免錯誤資訊 |
| 知識擴充責任 | 未來由同事協作，需建立多人可用的 SOP |
| 最優先解決問題 | **搜尋精準度不足**，這是第一優先 |

## 三、工作計畫

> **核心目標：讓新進人員查到正確的公安檢查知識，查不到就明確告知**

---

### 階段 A｜搜尋精準度（最優先）

#### A1. 建立黃金測試清單
- 列出 30–50 筆「標準查詢」，記錄預期結果與預期為空的情境
- 格式：

  | 查詢 | 預期結果 | 類型 |
  |------|---------|------|
  | 桃園市 H2 書表整理 | BS-TYNL | 應有結果 |
  | 台北市 水族箱 | （空） | 應為空 |
  | 新北市 H2 安全梯 | WKB-019 | 應有結果 |

- **誰來做**：先由你整理 10–20 筆最常遇到的查詢，後續同事補充

#### A2. 補強搜尋命中率
目前已知需改善的情境：

- **錯字容錯**：「申博書」→「申報書」、「安全梯」→「安全梯間」
- **縮寫對應**：「公安」→「公共安全申報」
- **口語查詢**：「要附什麼文件」→ 附件類條目
- **多詞同義**：「退件」= 「補件」= 「不予受理」

#### A3. 使用者回饋入口
- 「找不到」畫面加入「這個問題我需要答案」按鈕
- 收集到 Google Form 或直接開 GitHub Issue
- **目的**：讓新進人員主動反映知識庫缺口，取代猜測

---

### 階段 B｜知識擴充 SOP（協作基礎）

#### B1. Ragic → 知識庫的轉換流程

Ragic 是第一階段的資料來源，轉換 SOP：

1. 從 Ragic 匯出目標條目
2. 依標準格式改寫為知識卡片
3. 加入 `knowledgeBaseEntries.ts`
4. 用測試清單驗證可被搜尋命中
5. push → 自動部署

#### B2. 知識條目標準格式

每筆條目結構：

```
**【縣市】類別　主題標題**

**問：** 口語化的使用者問題

**判斷結論：** 一句話結論

（若需清單，換行列點）
1. 項目一
2. 項目二

**實務說明：** 背景說明、例外情形、注意事項

**缺失認定標準：** （若適用）
**禁止事項：** （若適用）

_WKB-XXX　類組：XX　階段：XX　審件風險：高/中/低_
```

#### B3. 優先擴充範圍

| 優先級 | 主題 | 說明 |
|--------|------|------|
| 🔴 高 | 各縣市 H2 附件清單 | 各地要求差異大，新人最常問 |
| 🔴 高 | 常見缺失認定（安全梯、消防、電梯） | 現場檢查核心知識 |
| 🟡 中 | 各縣市申報流程差異 | 跨縣市業務人員必備 |
| 🟡 中 | 改善計畫書格式與退件原因 | 退件風險高，需明確說明 |
| 🟢 低 | 法規條文對應 | 進階查詢，優先級較低 |

#### B4. 同事協作規範

- 新增條目前，先確認該問題在 Ragic 是否已有明確答案
- 若答案有縣市差異，需分開建立條目（不混用）
- 每次新增後，需實際搜尋驗證可命中
- 有疑義的答案需標記來源（電話確認、公文、法規條文）
